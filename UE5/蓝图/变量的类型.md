#### 命名

在虚幻引擎中，“命名 (Name)”（在 C++ 底层称为 `FName`）是一种专门为**极速查找和标识**而设计的特殊字符串类型。

它的核心作用可以概括为一句话：**它是引擎内部用来“贴标签”和“找东西”的极速 ID 卡。**

理解它的最佳方式，是把它和普通的“字符串 (String)”以及“文本 (Text)”区分开来。这三者是 UE 中处理文字的“三剑客”，职责完全不同。

**1.为什么不直接用 String？（底层的秘密）**

从 C++ 的底层架构来看，普通的字符串比较（比如判断 `"Player_Hand"` 是否等于 `"Player_Hand"`）需要逐个字符进行比对（`strcmp`），这在游戏每秒运行 60 帧、每帧要查找成千上万次骨骼或变量名的场景下，性能开销极其恐怖。

**`FName` 的解决方案：全局字符串表 (String Pool / Hash)**

- 当你创建一个 Name 变量 `"WeaponSocket"` 时，引擎不会在内存里存一串字符。它会把这个词存进一个全局的查找表中，并给它分配一个唯一的**整数 ID**（比如 `10042`）。
- 当引擎需要判断两个 Name 是否相等时，它实际上只是在**比较两个整数是否相等**。这在 CPU 层面是瞬间完成的。
- **特性：** 它不区分大小写（`MyName` 和 `myname` 是同一个 ID），并且它是**不可变的 (Immutable)**（你不能像 String 那样截取它的前三个字母，或者在后面追加字符）。

**2.“命名 (Name)” 的主要使用场景**

正是因为它的“极速比对”特性，它在引擎中专门用于那些**需要高频查找的标识符**：

- **骨骼与插槽 (Bones & Sockets)：** 当你把一把剑挂在角色的手上时，你填写的插槽名字（如 `"hand_r_socket"`）就是 Name 类型。
- **数据表行名 (Data Table Row Names)：** 当你从配置表中读取第 `105` 号怪物的属性时，那个行标签（Row Name）必须是 Name 类型。
- **材质参数查找：** 当你在蓝图或 C++ 中动态改变材质的发光强度时，传递的参数名（如 `"EmissiveMultiplier"`）是 Name 类型。
- **组件与 Actor 标签 (Tags)：** 你可以给场景里的陷阱打上 `"FireTrap"` 的 Tag，这种用于碰撞检测和逻辑筛选的标签也是 Name。
- **动画与蒙太奇：** 播放特定段落的动画（Section Name）或动画通知（Anim Notify）。

**3. 虚幻“文字三剑客”对比总结**

| **数据类型**        | **底层对应** | **核心定位**             | **性能**                      | **适用场景**                                            |
| ------------------- | ------------ | ------------------------ | ----------------------------- | ------------------------------------------------------- |
| **命名 (Name)**     | `FName`      | **内部标识与极速查找**   | **极快** (整数比较)           | 骨骼名、标签(Tag)、材质参数名、数据表行名。             |
| **字符串 (String)** | `FString`    | **动态处理与逻辑拼接**   | 较慢 (字符比较，内存分配)     | 拼接路径、处理纯逻辑文本、从服务器接收的 JSON 数据。    |
| **文本 (Text)**     | `FText`      | **玩家可见的本地化显示** | 复杂 (包含本地化和格式化逻辑) | UI 界面上的对话、物品名称、任务描述（支持多语言翻译）。 |

**避坑指南**

1. **不要混用：** 如果你要在屏幕的 UI 上显示角色的名字，绝对不要用 `Name`，必须用 `Text`。因为 `Name` 无法做多语言翻译。
2. **尽量少做转换：** 把 `String` 转换为 `Name` 是一个相对耗时的操作（因为引擎需要去全局表里计算 Hash 并查表）。如果你在 `Event Tick` 里每帧都在把一个动态 String 转成 Name 去查找，会导致性能问题。最好提前将其存为一个 Name 变量。
